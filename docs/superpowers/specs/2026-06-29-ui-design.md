# Intuitive UI — Cinematic Camera Screen — Design Spec

**Date:** 2026-06-29
**Mini-project:** #3 of the expexp revival program
**Status:** Approved for planning

## Program context

Mini-projects #1 (orientation), #4 (alignment), and #2 (color/blend, PR open) are done. This is
#3 (intuitive UI). After this: #5 ship to App Store. This branch (`mini-3-ui`) is stacked on
`mini-2-color` so it includes the filmic-average blend.

## Goal

Redesign the single camera screen to be **clean and beautiful**, make the **multi-exposure flow
obvious**, and add **live shooting guidance** — without cluttering the screen with creative
controls (looks/modes stay quiet; smart defaults already chosen). Direction "C · Cinematic" from
the mockups.

## Naming (working language)

Replace the vague "Magic / Snap Align" toggle with a **Raw vs Lock** distinction, where the lock
names itself by camera:
- **Raw** — alignment off; pure, what-you-shoot-is-what-you-get double exposure.
- **Edge Lock** — back camera; anchors the static structure (cityscape).
- **Face Lock** — front camera; anchors the face (selfie swirl).

These are working names; exact wording can still be refined and is not a blocker.

## Design

### Layout (cinematic-minimal)
Full-screen preview + ghost overlay (unchanged). Over it:
- **Shutter** — white circle, center-bottom; shows `+` once a stack is in progress.
- **Lock ring** (around the shutter) — a **scalar, smoothed** ring that fills/brightens amber→green
  as the device steadies onto the first frame; solid green + glow = locked. Shown only in Lock mode
  with a stack in progress (there must be a first frame to lock against). Absent in Raw.
- **Exposure count** — a small row of dots (e.g. ●●○○) plus a short label ("2 exposures"),
  positioned **low on screen** (just above the shutter), clear of the Dynamic Island.
- **Flip camera** (left) and **Save** (right) — quiet, faded side buttons; Save brightens to a
  green ✓ once ≥1 exposure exists; tapping Save keeps the transient saved-thumbnail (tap → Photos).
- **Discard stack** — a subtle control that appears only when a stack exists (clears the buffer).
- **Ghost opacity** — a slim slider that fades in on tap and auto-hides; sensible default otherwise.
- **Mode** — a quiet control at the bottom toggling Raw ⟷ Lock; its label reads
  **Raw / Edge Lock / Face Lock** by context.
- **Looks** (Neutral/Filmic/Moody) — behind a quiet `⋯` button; Filmic stays the default. Minimal,
  since creative controls are intentionally not front-and-center.

### Live-guidance tech: gyro-driven lock ring
The lock ring is driven by **Core Motion device attitude**, not continuous video registration:
- On the first capture of a stack, record the device's reference attitude.
- While composing the next exposure, read the current attitude, compute the angular deviation from
  the reference, map it to a 0…1 "lock progress" (1 = aligned), and **smooth it (EMA)** to avoid
  jitter; snap to a clear "locked" state within a small tolerance.
- This is cheap (no per-frame image processing), smooth, and immune to moving subjects — the device
  motion is what the user controls to steady the static structure.
- The save-time **optical** alignment (mini-project #4) is unchanged; the gyro only powers live
  guidance.

### Code structure
`ContentView` has grown into one large body; split the camera UI into focused SwiftUI subviews so
each piece is small and testable:
- `ContentView` — top-level: preview, permission state, overlays, wiring.
- `CameraControlsView` — the bottom controls cluster (shutter, flip, save, discard, mode, opacity,
  looks).
- `ShutterButton` — the shutter + lock ring.
- `ExposureCountView` — the dots + label.
- `LockRingView` — the scalar ring (reads `lockProgress`).
- `LockMonitor` (Phase B) — wraps `CMMotionManager`; publishes `lockProgress`. Owned/surfaced via
  `CameraModel`.

`CameraModel` gains light UI-facing state: a `modeLabel` (Raw / Edge Lock / Face Lock), a
`lockProgress: Double`, and `startLockReference()` / `stopLockMonitoring()` hooks. The existing
`isAlignmentEnabled` becomes the Raw/Lock toggle (Raw = off).

## Phased implementation

- **Phase A — layout & flow (pure SwiftUI):** the cinematic layout, exposure-count dots, restyled
  shutter/flip/save, discard control, opacity-on-tap, Raw/Edge-Lock/Face-Lock labeling, looks behind
  `⋯`. The lock ring is present visually but inert (neutral). Fully testable on device.
- **Phase B — gyro lock ring (adds Core Motion):** `LockMonitor` drives the scalar lock ring from
  device attitude; reference captured at first shot; smoothed; "locked" state. Test the live
  guidance.

## Non-goals

- Continuous **optical** lock-meter (gyro replaces the need).
- A prominent creative-controls panel (looks/modes stay quiet).
- Any change to capture, alignment, blend, orientation, or save *behavior* beyond UI/labels and the
  new gyro guidance.
- Onboarding/tutorial screens (the layout itself should make the flow obvious).

## Testing (on device — camera does not run in the Simulator)

Simulator build must compile first. On a physical iPhone:

Phase A:
1. Multi-exposure flow reads clearly: take 1–4 shots; the **count dots** update and sit low,
   clear of the island; `+` appears on the shutter.
2. **Save** brightens once a shot exists; saving shows the thumbnail and clears the stack;
   **Discard** appears only with a stack and clears it.
3. **Mode** label reads **Raw** when off, **Edge Lock** (back) / **Face Lock** (front) when on, and
   flips correctly on camera switch.
4. **Opacity** slider fades in on tap and adjusts the ghost; screen stays calm otherwise.
5. **Looks `⋯`** reveals the three looks; selecting changes the live ghost (WYSIWYG).
6. No regressions: orientation, alignment (scene/face), filmic blend all still behave.

Phase B:
7. With a stack started in **Lock** mode, the **ring fills/brightens** as you steady the phone and
   goes **solid green** when aligned; it's smooth (no jitter).
8. Moving a subject through the frame does **not** disturb the ring (gyro-based).
9. In **Raw** mode the ring is absent; on the front camera it behaves sensibly (Face Lock).

## Risks & mitigations

- **Gyro lock mapping** (tuning deviation→0…1 and the "locked" tolerance): values are tunable on
  device; the ring is guidance, not a gate, so imperfect tuning is low-risk.
- **Layout churn across device sizes / safe areas:** use SwiftUI safe-area-aware layout; verify the
  count dots clear the Dynamic Island on the test device.
- **Scope creep into creative controls:** keep looks behind `⋯`; defer anything beyond the three
  approved looks.
